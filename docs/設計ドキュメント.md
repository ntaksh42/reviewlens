# ReviewLens 設計ドキュメント（やさしい解説版）

このドキュメントは、ReviewLens という拡張機能が「何をするものか」「どういう
しくみで動いているか」を、**プログラマーでなくても読めるように**まとめたもの
です。専門用語が出てくるときは、その都度かんたんな言葉で説明します。

> 技術的に踏み込んだ説明（部品ごとのソースコードや設計判断の根拠）は
> [`ARCHITECTURE.md`](./ARCHITECTURE.md) にあります。本書はその前段として、
> 全体像をつかむための読み物です。

---

## 1. ひとことで言うと

**ReviewLens は、「コードの修正チェック（レビュー）」を VS Code の中だけで、
すばやく・キーボード中心で行えるようにする道具です。**

- **VS Code** … プログラマーが文章（プログラム）を書くための高機能なエディタ。
  Microsoft 製で、世界中で広く使われています。
- **拡張機能（extension）** … VS Code に後から追加できる「アプリ」のようなもの。
  ReviewLens はその一つです。
- **Azure DevOps（ADO）** … Microsoft が提供する、チーム開発のための
  クラウドサービス。「誰がどこを直したか」を管理し、修正内容を提案・承認する
  場所でもあります。ReviewLens はこの ADO とやり取りします。

### レビューとは？

複数人で文章（プログラム）を書くとき、ある人が書いた修正案を、別の人が
「ここ大丈夫かな？」と読んで確認し、OK なら承認、気になれば指摘コメントを
付ける——この一連の作業を **レビュー** と呼びます。修正案のまとまりを
**プルリクエスト（Pull Request、略して PR）** と言います。

ReviewLens は、この「PR をレビューする」作業を専門に助ける道具です。

---

## 2. なぜ作ったのか（解決したい不満）

これまで PR のレビューは、たいてい **Web ブラウザ** で行われていました。
ただ、ブラウザでのレビューには次のような不便がありました。

| ブラウザでの不便 | ReviewLens での解決 |
|---|---|
| エディタとブラウザを行き来する必要がある | レビューもコードも **VS Code の中だけ** で完結 |
| マウス操作が多くて疲れる | **キーボードだけ** でファイル間・変更箇所間を移動できる |
| 「この関数、他のどこから呼ばれてる？」が分かりにくい | **影響分析** で「巻き込み範囲」を自動で洗い出す |
| 修正されたファイルしか深く調べられない | **ローカルレビュー** で実物のコードとして隅々まで調査できる |

特に最後の **影響分析（Impact analysis）** が、この道具のいちばんの差別化
ポイント（ウリ）です。詳しくは [第7章](#7-この道具のウリ影響分析) で説明します。

## 3. できること（機能の一覧）

VS Code の左端にある縦長のバー（アクティビティバー）に ReviewLens のアイコン
（🔀 プルリクエストのマーク）が追加されます。それを押すと、3 つのパネルが
開きます。

```
┌────────────── VS Code の画面 ──────────────┐
│ [🔀]  ┌─ Pull Requests ─────────┐          │
│ アイ  │ プロジェクトA            │  ← ①PR一覧 │
│ コン  │  └ リポジトリX           │          │
│ バー  │     └ #123 ログイン修正  │          │
│       ├─ Changed Files ─────────┤          │
│       │ #123 ログイン修正        │  ← ②変更 │
│       │  ✓ login.ts             │   ファイル │
│       │    auth.ts              │          │
│       ├─ Impact ────────────────┤          │
│       │ validateUser            │  ← ③影響  │
│       │  ⚠ checkout.ts:42       │   分析    │
│       └────────────────────────┘          │
└────────────────────────────────────────────┘
```

### ① Pull Requests（PR 一覧）
- 今レビュー待ちになっている PR の一覧を、**プロジェクト → リポジトリ** の
  順にグループ分けして表示します。
  - **リポジトリ** … プログラムの保管庫。1 つのソフトの全ファイルが入った
    フォルダのようなもの。
  - **プロジェクト** … リポジトリをいくつかまとめた、もっと大きな単位。
- 上部の **フィルター**（🔍）に文字を打つと、タイトル・作成者・リポジトリ名
  などで一覧をしぼり込めます。

### ② Changed Files（変更されたファイル）
- 一覧から PR を 1 つクリックすると、その PR で **変更されたファイル** が
  ここに並びます。
- ファイルをクリックすると、**修正前 ↔ 修正後** を左右に並べた比較画面
  （**差分／diff**）が開きます。
- **既読（Viewed）** … 確認し終わったファイルに ✓ を付けられます（PR ごとに
  記憶されます）。
- このパネル上部のボタンから、後述の **投票・完了**、**ローカルレビュー**
  も操作します。

### ③ Impact（影響分析）
- この PR で変更された関数が、**他のどこから呼び出されているか** を一覧化し、
  **「この PR では直されていない呼び出し元」** に ⚠ マークを付けて警告します
  （＝うっかり壊しやすい場所）。

### そのほかの機能
- **コメント** … 差分画面の各行に、指摘コメントを付けられます。書いた内容は
  ADO に送られ、相手にも見えます。返信や「解決済み」への変更もできます。
  キーボードだけで、`Ctrl+;` でコメント開始 → `Ctrl+Enter` で送信できます
  （Mac は `Cmd`）。
- **投票（Vote）と仕上げ** … レビューの結論を記録します。「承認 / 提案つき
  承認 / 作者の対応待ち / 却下 / 取り消し」。さらに PR の **完了（マージ＝
  本体へ取り込み）** や **中止（取りやめ）** もここから行えます。取り返しの
  つかない操作なので、実行前に確認が出ます。
- **キーボード移動** … 変更ファイル間（`Shift+Alt+]` / `[`）や、差分の中の
  変更箇所間（`Alt+↓` / `↑`）をキーだけで移動できます。

## 4. 全体のしくみ（建物にたとえると）

ReviewLens の中身は、役割ごとに **4 つの階層** に整理されています。これは
「混ざると分かりにくくなるもの」をきっちり分けるための工夫です。建物の
フロアにたとえると分かりやすいです。

```
  ┌─────────────────────────────────────────────┐
  │  ④ UI 層（受付・店頭）                          │
  │     画面表示・ボタン・キー操作。利用者と直接やりとり  │
  ├─────────────────────────────────────────────┤
  │  ③ App 層（店長）                              │
  │     「PRを開く」「投票する」など、やりたいことの段取り │
  ├─────────────────────────────────────────────┤
  │  ② Infra 層（外部とつなぐ配送・倉庫）             │
  │     Azure DevOpsとの通信・ログイン・保存・git操作    │
  ├─────────────────────────────────────────────┤
  │  ① Domain 層（社内ルールブック）                 │
  │     「PRとは」「コメントとは」という言葉の定義だけ     │
  └─────────────────────────────────────────────┘
```

| 層 | たとえ | 役割 | フォルダ |
|---|---|---|---|
| ④ UI | 店頭・受付 | 画面に出す、ボタンを受け付ける、キー操作を処理する | `src/ui/` |
| ③ App | 店長 | 「PR を開く」「コメントする」などの段取りを指揮する | `src/app/` |
| ② Infra | 配送・倉庫 | 外の世界（ADO・git・保存先）と実際にやりとりする | `src/infra/` |
| ① Domain | ルールブック | 「PR」「コメント」などの“言葉の意味”だけを定義する | `src/domain/` |

**なぜ分けるのか？** たとえば将来 Azure DevOps が GitHub に変わっても、
作り変えるのは ②（外部とつなぐ部分）だけで済み、④画面や③段取りはそのまま
使える——というように、**変更の影響を一部に閉じ込められる** からです。
上の層は下の層に頼りますが、下の層は上の層を知りません（受付は店長を頼れる
が、ルールブックは受付の存在を知らない、という一方通行）。

## 5. 使うときの流れ（裏側で何が起きているか）

実際の操作を例に、4 つの層がどう連携するかを追ってみます。

### 準備：サインインと設定
1. 設定画面で、自分の組織の URL・プロジェクト名などを入力します。
2. **「ReviewLens: Sign in (PAT)」** を実行し、**PAT** を入力します。
   - **PAT（Personal Access Token）** … パスワードの代わりに使う「合鍵」の
     ような文字列。これがあると ReviewLens があなたの代理で ADO にアクセス
     できます。**金庫（VS Code の SecretStorage）に安全に保管** され、画面や
     ログに出ることはありません。

### レビューの流れ
```
[あなた]                [UI層]            [App層]          [Infra層]        [Azure DevOps]
   │  PRをクリック         │                  │                 │                 │
   ├────────────────────▶│  「開いて」        │                 │                 │
   │                     ├────────────────▶│  「ファイル一覧と  │                 │
   │                     │                  │   コメントを取得」 │                 │
   │                     │                  ├───────────────▶│  PR #123の中身は？│
   │                     │                  │                 ├───────────────▶│
   │                     │                  │                 │◀───────────────┤
   │                     │◀────────────────┤◀───────────────┤  変更ファイル＋  │
   │  ②に一覧が表示される  │  画面に並べる     │                 │   コメント       │
   │◀────────────────────┤                  │                 │                 │
```

1. **PR をクリック**（UI 層が受け取る）
2. UI 層が App 層に「この PR を開いて」と依頼
3. App 層が Infra 層を通じて ADO に問い合わせ、**変更ファイルの一覧** と
   **既存コメント** を同時に取りに行く（待ち時間を短くするため並行で取得）
4. 受け取った結果を UI 層が画面（②）に並べる
5. ファイルをクリックすると、修正前・修正後の中身を ADO から取り寄せて
   **差分画面** を表示。既存コメントも該当行に重ねて表示する

> **効率化の工夫**：一度取り寄せたファイルの中身は **記憶（キャッシュ）** して
> おき、行き来しても取り直さないようにしています。ADO への接続も使い回して、
> 毎回の待ち時間を減らしています。

## 6. ローカルレビュー（実物のコードとして調べるモード）

通常の差分表示は「修正前・修正後の中身を ADO から借りてきて見せている」
だけなので、**その場で表示している以外のことはできません**（例：「この関数の
定義へジャンプ」ができない）。

そこで **ローカルレビュー** を ON にすると、ReviewLens は **あなたのパソコンに
ある、そのソフトの複製（ローカルクローン）** を使い、PR の修正後の状態を
**本物のファイルとして** 取り出します。すると——

- **定義へジャンプ**（この関数の中身はどこ？）
- **参照を探す**（これはどこから使われてる？）
- **横断検索（grep）**、**隣のファイルを開く**

といった、ふだんプログラミングで使える便利機能がレビュー中にも全部使える
ようになります。

### 安全のための工夫（あなたの作業を絶対に汚さない）

ここがよく考えられている点です。普通にやると「自分の作業中のファイルが
PR の状態に書き換わってしまう」危険があります。それを避けるため：

- **worktree（ワークツリー）** という git の仕組みを使い、**別の場所に
  “使い捨ての作業コピー”** を作ります。あなたが今開いているファイルや
  作業中のブランチには **一切触れません**。
  - **git** … ファイルの変更履歴を管理する、開発の必須ツール。
  - **ブランチ** … 「本流と分けて作業するための枝分かれ」。
- 取り出したファイルは **その回のあいだ読み取り専用** にします。うっかり
  編集してコメントの位置がズレたり、作業コピーが汚れたりしないためです。
- 作った作業コピーは **使い回し**（同じ修正なら再利用）し、増えすぎたら
  **古いものから自動でお片付け**します（既定で 5 個まで保持）。この個数は
  設定 `reviewlens.localWorktreeLimit` で変えられます。

ローカルレビューが必要なければ ON にしなくて構いません。軽い ADO だけの
差分表示が **既定（デフォルト）** で、ローカルレビューは **任意（opt-in）** です。

## 7. この道具のウリ：影響分析

レビューでいちばん怖いのは、**「直した関数のせいで、別の場所が静かに壊れる」**
ことです。修正された場所は目に入りますが、**その関数を使っている“呼び出し元”**
は画面に出てこないため、見落としがちです。

**影響分析**は、これを自動で洗い出します。

```
   この PR で変更された関数：validateUser()
        │
        ├─ どこから呼ばれている？（VS Codeの解析エンジンに尋ねる）
        │
        ▼
   呼び出し元の一覧：
     ✓ login.ts:88      ← この PR で一緒に直されている（たぶん安心）
     ⚠ checkout.ts:42   ← この PR では直されていない！（要確認＝巻き込み範囲）
     ⚠ report.ts:15     ← 同上
```

- VS Code に元から備わっている **コード解析エンジン（言語サーバー）** に
  「この関数は誰から呼ばれている？」と問い合わせ、結果を集計します。
- **この PR で直されていない呼び出し元** に ⚠ を付けます。ここが
  **「巻き込み範囲（blast radius）」**——レビュアーがいちばん見落としやすく、
  かつ壊れやすい場所です。
- ⚠ をクリックすれば、その箇所へすぐ飛んで確認できます。

ローカルレビューが ON のときは、PR の修正後の実物コードに対して解析するので、
より正確な結果が得られます。

## 8. データの保存場所と安全性

ReviewLens が「覚えておくこと」と、その置き場所です。

| 覚えること | 置き場所 | 説明 |
|---|---|---|
| PAT（合鍵） | **SecretStorage（金庫）** | VS Code が暗号化して安全に保管。画面・ログには出さない |
| 既読チェック（✓） | パソコン内のローカル記憶 | PR ごとに「どのファイルを見たか」を記憶 |
| コメントの目印 | パソコン内のローカル記憶 | コメントが付いた行の“見た目”を覚えておく（後述） |
| 作業コピー（worktree） | 拡張専用の保管フォルダ | ローカルレビュー用。古いものは自動削除 |
| コメント本体・投票・承認 | **Azure DevOps（クラウド）** | チーム全員に共有される正式な記録はすべて ADO 側 |

ポイントは、**チームで共有すべき正式なもの（コメント・投票・承認・完了）は
すべて ADO に保存** され、ReviewLens 自身のパソコン内メモは「あなたの手元の
快適さ」のための補助情報だけ、という切り分けです。

### コメントが消えない工夫（再アンカリング）

PR は何度も更新されます。すると、せっかく付けたコメントの行が上下にズレて
しまうことがあります。ReviewLens は、コメントを付けたとき **その行の文字内容を
こっそり覚えておき**、後で行がズレても **同じ内容の行を探し直して** コメントを
正しい位置に貼り直します。動いた場合は `↕`、見つからなかった場合は `⚠` の
印で知らせます。これにより「指摘した場所が分からなくなる」事故を防ぎます。

## 9. まだ無い機能（今後の予定）

- **修正提案（Suggested edits）** … 「ここをこう直して」と具体的なコードを
  提案する機能。ADO 側に対応する仕組みがないため、今は見送り。
- **AI 機能** … PR の自動要約・リスク判定・変更内容の説明など。将来の
  フェーズで検討予定（設計上、その置き場所だけは確保済み）。

## 10. 用語ミニ辞典

| 用語 | やさしい説明 |
|---|---|
| VS Code | プログラムを書くための高機能エディタ |
| 拡張機能 | VS Code に後付けできるアプリ。ReviewLens もこれ |
| Azure DevOps（ADO） | 修正案の管理・承認を行うクラウドサービス |
| プルリクエスト（PR） | 「この修正を取り込んでほしい」という提案のまとまり |
| レビュー | 他人の修正案を読んで確認し、指摘・承認する作業 |
| 差分（diff） | 修正前と修正後を見比べた表示 |
| リポジトリ | 1 つのソフトの全ファイルが入った保管庫 |
| ブランチ | 本流と分けて作業するための枝分かれ |
| マージ | 修正案を本流に取り込むこと（＝PR の完了） |
| git | ファイルの変更履歴を管理する開発ツール |
| worktree | git で作る「使い捨ての作業コピー」 |
| PAT | パスワード代わりの合鍵となる文字列 |
| キャッシュ | 一度取り寄せたものを再利用するための一時記憶 |
| 言語サーバー | コードを解析して定義や呼び出し元を教えてくれる仕組み |
| 巻き込み範囲（blast radius） | ある修正が壊しうる、波及先の範囲 |
